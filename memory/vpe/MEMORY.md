# Global Rules

- **Read-only git is allowed without asking** ‚Äî diff, show, status, log, blame, and assembling read-only artifacts from them (a diff/review package, a change ledger). Anything that **writes or changes state I NEVER run** ‚Äî you run those yourself: add, commit, push, checkout, reset, rebase, merge, tag, branch edits. See [[feedback_never_commit_push]] and [[feedback_gh_read_only_never_comment_as_user]].

# Feedback

- [Don't over-engineer ‚Äî lead with the simplest alternative](feedback_prefer_simplest_alternative.md) ‚Äî check existing flags/published-data/off-the-shelf tools before building new nodes/code; preference order existing-config > published-topics > off-the-shelf > small reuse > new code
- [Never run git commit or git push](feedback_never_commit_push.md) ‚Äî even when read-only git is authorized, user runs commit/push themselves
- [gh + git are read-only; never post as the user](feedback_gh_read_only_never_comment_as_user.md) ‚Äî no PR comments, review replies, merges, pushes, or any `gh`/`git` write command, even when read-only access is granted
- [Never mention the assistant in commits or PRs](feedback_no_assistant_attribution_in_commits.md) ‚Äî no Co-Authored-By, no generation footer, no reference to Claude/calude/AI/CLAUDE.md in commit messages or PR descriptions; assistant config files are gitignored
- [Always plan first ‚Äî write to the plan file before editing](feedback_always_plan_first.md) ‚Äî even outside formal plan mode, write the proposed edits to the plan file before the first Edit/Write call so the user can review scope
- [Scratch/tmp under INTERNAL/tmp/, never /tmp](feedback_tmp_under_internal.md) ‚Äî redirect run logs/debug dumps to INTERNAL/tmp/<subfolder>/ (gitignored, visible, deletable); don't litter /tmp
- [uv toolchain preferred for Python work](feedback_uv_toolchain_preferred.md) ‚Äî always `uv env`/`sync`/`add`/`run` + `.venv`; don't prefix uv calls with `env -u` env-stripping dances
- [Commit messages: Jira tag + no backticks, concise](feedback_commit_message_format.md) ‚Äî lead with repo's ticket tag (e.g. V5-136:); backticks/$() trigger shell substitution on paste and break commits; short subject, body only for non-obvious why
- [Run tests before updating docs, never the other way around](feedback_test_before_docs.md) ‚Äî sequence is code ‚Üí tests ‚Üí colcon build ‚Üí README/CLAUDE.md/handover docs; doc updates wait until the build is green
- [SDD needs a Head-of-Robotics logic overseer + a code-quality checker](feedback_overseer_agent.md) ‚Äî for subagent-driven-development or ANY multi-subagent workflow, ALWAYS run two dedicated review roles: a logic overseer with the "Head of Robotics & 3D Perception" persona (math/frames/observability/numerical correctness) + a code-quality checker; the logic overseer is the cross-cutting catch-net for the final sign-off
- [All workspace docs authored under "Adil Shaikh"](feedback_doc_authorship.md) ‚Äî handover/spec/design docs use Adil Shaikh in the Author field, never "VPE Development Team" or similar placeholders; applies to v5_vehicle_pose_estimator workspace only
- [Perception changes need full e2e + visual verification before "done"](feedback_perception_e2e_verification.md) ‚Äî unit tests + colcon build are necessary but not sufficient; always run `run_investigation.sh` against `INTERNAL/booth_test_slow_bag` and inspect the result via Foxglove on a `--record` MCAP (in-loop PNG snapshots removed) before declaring a perception change complete
- [Run the e2e after EVERY pipeline change, not last](feedback_run_e2e_after_every_change.md) ‚Äî validate each live-pipeline edit in the e2e immediately (edit‚Üíunit‚Üíbuild‚Üí`run_investigation.sh`‚Üíinspect), don't batch edits and defer the e2e to the end
- [Recognise the car in snapshots ‚Äî no generic cluster interpretations](feedback_recognize_car_in_snapshots.md) ‚Äî when reading phase snapshots, identify the actual vehicle (position, orientation, dimensions) and compare the OBB to it concretely; do not describe the cluster as if it's the ground truth without first locating where the car actually is
- [Never reindex a rosbag without explicit permission](feedback_never_reindex_bags_without_permission.md) ‚Äî `ros2 bag reindex` overwrites `metadata.yaml` with whatever the storage plugin can currently read; ask first, even on a copy
- [Always debug with real bag values, never synthetic](feedback_debug_with_real_bag.md) ‚Äî instrument the real node + re-run the `INTERNAL/` bag; a constant-velocity synthetic hid the start-from-rest acceleration lag the real bag showed
- [Always source .venv before building/running](feedback_source_venv_before_build.md) ‚Äî `source .venv/bin/activate` first, then ROS; open3d/small_gicp are venv-only, else ros2 launch uses system Python and tracking_node dies on `No module named 'open3d'` with empty metrics.csv
- [Always ask, never assume ‚Äî fact-check to 100% before asserting](feedback_always_ask_never_assume.md) ‚Äî NEVER state an interpretation/conclusion as fact until verified against real data/code; treat every analysis as a hypothesis not a verdict; no "this works"/"dead end"/"root cause is X" from one run; one contradicting data point ‚Üí STOP and re-verify, don't defend (a whole black-bag debug session of premature conclusions had to be overturned one by one)
- [Always run PRODUCTION e2e, never CV-pinned](feedback_always_production_config_e2e.md) ‚Äî run the tracking e2e in the deployed adaptive-filter config (no env overrides); CV-pinned isolation numbers don't reflect production, the user can't infer from them ("ALWAYS RUN PRODUCTION RUNS")
- [Always run a FRESH e2e for analysis; never reuse existing dumps](feedback_always_fresh_e2e.md) ‚Äî every accuracy/truth/drift analysis runs a new `run_investigation.sh`, then analyses that run; don't point tools at older diagnostics_runs/ dumps
- [Always pass --record on every e2e run](feedback_always_record_e2e.md) ‚Äî every `run_investigation.sh` invocation (sweeps + quick checks included) needs `--record` so the user can inspect the MCAP in Foxglove; still NEVER `--cad-eval`

- [VPE brainstorm captures live under docs/brainstorms/](project_vpe_brainstorm_location.md) ‚Äî grill-me/discovery docs go to docs/brainstorms/YYYY-MM-DD-slug.md (not the grill-me default, not the plan file); add an Outcome section when work lands
